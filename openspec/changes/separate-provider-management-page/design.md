## Context

当前设置页只有一个 `providers` 分类，它同时承担三类职责：内置供应商 API Key 配置、供应商说明文档、自定义供应商的新增与编辑。随着自定义 OpenAPI 供应商已经支持多供应商、多模型配置，原有“长表单堆叠卡片”方式不适合快速查看已有配置，也不符合“添加一个供应商 / 编辑一个供应商”的操作心智。

现有运行时链路已经基于 `customProviders -> runtimeRegistry -> ModelParamsControls -> Tauri providerRuntime` 打通，自定义供应商可直接参与生图选择。因此本次变更的重点应放在设置页信息架构与交互重构，而不是改动底层持久化或生成协议。

## Goals / Non-Goals

**Goals:**
- 在设置页左侧提供独立的“供应商”入口，专门管理自定义供应商。
- 将自定义供应商改为“列表页 + 弹窗新增/编辑”的管理模式。
- 让供应商列表行直接提供“编辑”“删除”操作，并对删除进行二次确认。
- 保留现有 `customProviders` 存储结构与生成运行时兼容性。
- 让用户在完成新增或编辑后，能继续在生图时选择这些自定义供应商与模型。

**Non-Goals:**
- 不改变当前 OpenAPI 兼容协议的请求/响应格式。
- 不调整 `customProviders` 的底层字段结构或 runtime registry 映射规则。
- 不在本次提案中新增“连通性测试”“健康检查”“异步任务协议”之类扩展能力。
- 不重做整个设置页视觉系统，只针对供应商管理信息架构做局部重构。

## Decisions

### 1. 设置页新增 `suppliers` 分类，左侧文案固定为“供应商”

自定义供应商管理将从现有 `providers` 分类中拆出，新增独立设置分类，例如 `suppliers`。左侧用户可见文案固定为“供应商”，对应独立页面。

保留现有 `providers` 分类，但将其职责收敛为“内置 API 配置 / 供应商说明”，避免两个入口同时承担“供应商管理”语义。推荐将旧分类文案调整为“API 配置”，用于承接 KIE、PPIO、FAL、GRSAI 的密钥配置和说明文档。

备选方案：
- 保持单页、只在原 `providers` 分类内部上下分区：被否决，因为不满足“单独提到左侧”的要求。
- 直接复用 `providers` 作为新页面并把内置配置挪走：被否决，因为会破坏现有所有内置 API Key 跳转入口。

### 2. 自定义供应商页面采用“列表页 + 弹窗编辑器”

供应商页面本身只负责展示：
- 页面标题与说明
- 顶部“添加供应商”按钮
- 已保存自定义供应商列表

供应商表单不再直接内嵌在页面中，而是拆成可复用的弹窗编辑器组件：
- `Add` 场景打开空白草稿
- `Edit` 场景打开当前供应商的副本草稿

这样可以复用当前已有字段定义和校验逻辑，同时让列表页保持扫描效率。

备选方案：
- 继续沿用当前内嵌卡片编辑：被否决，因为列表不可扫描、编辑成本高。
- 使用独立全页表单而不是弹窗：被否决，因为用户需求明确要求弹出窗口。

### 3. 弹窗内保存即完成供应商配置保存，不再依赖页面底部二次保存

“添加供应商”与“编辑供应商”的弹窗底部都提供明确的“保存”按钮。用户点击后，当前供应商草稿会先经过与现有实现一致的字段校验，再直接写回 `customProviders`。

这意味着供应商页采用“操作即保存”的交互，而不是复用设置页其它分类的全局底部保存按钮。这样可以避免“双重保存”的心智冲突，也更符合用户对弹窗表单的预期。

备选方案：
- 弹窗只更新本地草稿，仍需回到设置页点一次总保存：被否决，因为和需求中的“点击下方保存按钮保存”不一致，且容易误以为已生效。

### 4. 保存后的自定义供应商继续复用现有运行时链路

本次不修改以下契约：
- `customProviders` 的数据结构
- `runtimeRegistry` 对自定义供应商模型的聚合逻辑
- `ModelParamsControls` 中的供应商/模型二级级联逻辑
- `providerRuntime` 透传到 Tauri 的生成链路

因此新增或编辑供应商后，只要保存成功，对应供应商和已启用模型就必须继续出现在生图选择中；若某个已被引用的自定义模型失效，仍复用当前已有的默认模型回退策略。

### 5. 列表行展示“供应商名称 + 可用模型摘要 + 编辑 / 删除按钮”

供应商列表行采用轻量信息密度布局：
- 主标题：供应商名称
- 次级信息：已启用模型的显示名摘要，多个模型使用 `、` 连接
- 右侧操作：编辑按钮、删除按钮

当模型过多时，摘要允许截断，但应保持“有多少个可用模型”这一事实可感知，例如显示前若干个名称并保留数量信息。

删除按钮必须先进入二次确认流程，确认后才真正从 `customProviders` 中移除对应条目。删除后的列表刷新仍由 store 更新自动驱动，不允许要求用户回到设置页底部再点一次总保存。

## Risks / Trade-offs

- [供应商页采用即时保存，而其它设置页仍是延迟保存] -> 通过独立页面文案和弹窗保存按钮强化心智，并确保供应商页不再出现额外总保存按钮。
- [新增左侧分类后，现有 `openSettingsDialog({ category: 'providers' })` 跳转语义会变得含混] -> 为自定义供应商缺配置场景新增 `suppliers` 跳转，内置供应商缺 key 场景继续使用 `providers`。
- [模型摘要过长导致列表可读性下降] -> 摘要默认单行截断，并保留模型数量提示。
- [编辑或删除供应商后影响当前节点所选模型] -> 继续复用现有 runtime fallback 逻辑，避免节点进入不可生成状态。

## Migration Plan

本次不需要数据迁移。已有 `customProviders` 持久化数据保持原样，设置页只是更换展示入口与编辑方式。

上线步骤：
1. 新增 `suppliers` 设置分类和独立页面。
2. 将当前自定义供应商表单从 `providers` 页面迁出。
3. 复用现有校验与存储逻辑接入弹窗保存。
4. 调整生图端“去配置供应商”的跳转目标。

回滚策略：
- 如需回滚，只需恢复旧的内嵌式 `CustomProviderSection`，无需迁移用户数据。

## Open Questions

- 无。当前口径已经确定为“列表行直接显示删除按钮，但删除必须二次确认”，并继续沿用现有 runtime fallback 处理已被节点引用的旧模型。
