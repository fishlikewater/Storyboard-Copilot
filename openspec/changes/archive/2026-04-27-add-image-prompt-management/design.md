## Context

当前 `ImageEditNode` 在标题下方直接展示一个大文本框，节点本身没有任何可复用提示词入口；用户只能手动输入或从外部复制内容。与此同时，应用已经有一套稳定的用户级设置持久化机制：`useSettingsStore` 使用 Zustand `persist` 保存 API Key、下载路径、自定义供应商等跨项目配置。

这意味着提示词模板更适合作为“用户长期偏好”而不是“单个项目数据”。讨论过程中，入口和应用行为也已经完成收敛：

- 管理入口采用顶栏独立按钮，而不是设置页左侧分类。
- 模板点击行为采用“追加到末尾，并在中间自动插入空行”，而不是整段替换。

因此，本次设计需要同时修正入口架构和节点交互口径。

## Goals / Non-Goals

**Goals:**

- 为用户提供一套全局持久化的提示词模板管理能力，模板至少包含标题和提示词正文。
- 在顶栏设置按钮旁提供独立入口，点击后直接打开“提示词管理”弹窗。
- 在 `AI 图片` 节点中以快捷标题形式展示这些模板，点击即可把正文写入当前节点提示词框。
- 保持 `ImageEditNode` 现有模型选择、比例选择、参考图插入和生成链路不变。
- 让新增能力具备清晰的校验、空状态、删除确认、列表刷新和回归测试。

**Non-Goals:**

- 不在本次设计中加入提示词分类、标签分组、搜索、排序拖拽或导入导出。
- 不将提示词模板做成项目级数据，也不随项目文件一起持久化。
- 不扩展到 `StoryboardGenNode`、文本注释节点或其他非 `AI 图片` 场景。
- 不实现按光标位置插入模板、模板拼接规则配置或模板参数化。

## Approach Comparison

### 1. 顶栏独立按钮 + 独立提示词管理弹窗（推荐）

在顶栏设置按钮旁新增一个独立按钮，点击后直接打开“提示词管理”弹窗。弹窗只负责模板列表、添加、编辑和删除，不再经过设置页分类。

优点：

- 入口最短，最符合用户已确认的交互。
- 不再给 `SettingsDialog` 增加新的分类负担。
- 可以复用当前项目已经成熟的弹窗交互风格。

缺点：

- 顶栏会新增一个按钮，需要注意图标、文案和 hover 态的可辨识度。

### 2. 顶栏独立按钮 + 打开设置页中的“提示词”分类

顶栏新增按钮，但点击后仍打开 `SettingsDialog`，并自动定位到某个新加的“提示词”分类。

优点：

- 能最大化复用现有设置容器。

缺点：

- 用户虽然通过顶栏新按钮进入，但仍然需要接受“这其实还是设置页的一部分”的心智，和“独立管理入口”不完全一致。
- 会继续增加设置页信息密度。

### 3. 顶栏独立按钮 + 右侧抽屉

点击顶栏按钮后打开抽屉式面板，在右侧完成模板管理。

优点：

- 视觉上更轻，理论上适合边看边改。

缺点：

- 当前项目主交互语言是弹窗，不是抽屉。
- 需要额外设计抽屉的关闭、遮罩、滚动与嵌套弹层策略，超出这次需求的必要复杂度。

## Decisions

### 1. 提示词模板存入 `settingsStore`，作为全局用户偏好持久化

新增 `promptTemplates` 到 `useSettingsStore`，并配套独立的类型、归一化和校验辅助模块。模板字段至少包括：

- `id`：稳定标识
- `title`：显示在快捷标题上的短文本
- `content`：点击后带入节点提示词框的完整正文

选择全局设置而不是项目数据，原因是这批模板本质上是个人常用语料，预期会跨项目复用；放进设置持久化还能天然获得现有的本地存储和迁移机制。

### 2. 管理入口放在顶栏设置按钮旁，打开独立提示词管理弹窗

这次设计不再扩展 `SettingsCategory`。取而代之的是：

- `TitleBar` 在设置按钮旁新增一个独立的提示词管理按钮。
- `App.tsx` 维护专用的弹窗开关状态。
- 点击该按钮后，直接打开独立的提示词管理弹窗。

设置按钮继续只负责设置，提示词按钮只负责提示词模板管理，职责保持清晰。

### 3. 管理弹窗采用“列表页 + 二级编辑弹窗 + 删除确认”结构

提示词管理弹窗主体结构如下：

- 顶部标题与“添加提示词”按钮
- 下方已保存模板列表
- 列表行显示“标题 + 提示词内容摘要”
- 每行右侧提供“编辑”与“删除”按钮

点击“添加”或“编辑”时，打开二级编辑弹窗，字段只包含：

- 标题
- 提示词内容

编辑弹窗底部使用明确的“保存”按钮；保存成功后立即刷新列表。删除则必须经过二次确认，确认后立即从列表中移除。

### 4. 快捷标题显示在 `ImageEditNode` 标题下方、文本框上方，采用两行换行式布局

为了贴合用户给出的视觉预期，快捷标题区放在节点头部之后、提示词编辑区之前，使用现有 chip 风格做成一组可点击的短标签。布局要求：

- 仅当存在模板时显示
- 按保存顺序展示
- 允许换行
- 最多显示两行高度
- 模板过多时在快捷区内部滚动，不挤压下方大文本框

这样既能维持节点顶部的可读性，也能保证大文本框依然是主编辑区。

### 5. 点击快捷标题时，空文本直接写入；非空文本先插入空行再追加模板内容

节点内点击模板后的行为统一为：

- 当前提示词为空：直接写入模板内容
- 当前提示词非空：在原文本末尾插入一个空行，再追加模板内容

实现上需要同时更新：

- `promptDraft`
- `promptDraftRef`
- 节点数据中的 `data.prompt`

这样才能保证可见文本框、内部草稿和生成请求读取到的是同一份 prompt 数据。

### 6. 标题和正文都必须校验，标题在归一化后需要保持唯一

管理页保存前应至少校验：

- 标题非空
- 正文非空
- 标题去首尾空格后不重复

校验失败时，编辑弹窗保持打开，并在字段附近给出明确错误提示，不做模糊的全局失败提示。删除操作则始终要求用户进行二次确认。

## Risks / Trade-offs

- [顶栏新增一个按钮会增加入口密度] -> 通过清晰图标、tooltip 与靠近设置按钮的布局，保持用户对“全局管理入口”的理解连续性。
- [模板过多会挤压节点顶部空间] -> 快捷标题区限制为最多两行高度，并独立滚动。
- [追加式应用可能造成重复内容] -> 保持行为简单直接，不做自动去重；由用户显式点击触发，结果可预期。
- [全局模板不一定适合所有项目] -> 本次明确将其定义为用户级偏好；若未来出现强烈项目隔离需求，再单独设计项目级模板集。

## Migration Plan

1. 为 `settingsStore` 新增 `promptTemplates` 字段，并在 `persist` 迁移逻辑中为旧版本补默认空数组。
2. 在 `TitleBar` 与 `App.tsx` 中接入新的提示词管理入口与独立弹窗；旧版设置页结构不需要承接该能力。
3. 在 `ImageEditNode` 中接入模板快捷标题区；当模板数组为空时不显示快捷区，确保旧界面仍可正常使用。

回滚策略：

- 若需要回滚，只需移除提示词管理弹窗、顶栏入口和 `ImageEditNode` 的快捷标题区。
- 旧版本即使忽略新增的持久化字段，也不会影响原有设置读取。
