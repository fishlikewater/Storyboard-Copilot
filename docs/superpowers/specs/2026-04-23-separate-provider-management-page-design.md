# 自定义供应商独立管理页设计

## 1. 背景

当前分支已经完成了 OpenAPI 自定义供应商能力接入：

- 用户可以在设置页配置多个自定义供应商
- 每个供应商可以配置 `baseUrl`、`apiKey`、协议类型和多个模型
- 生图节点已经支持“供应商 -> 模型”的二级选择
- 运行时会根据所选自定义供应商自动切换到 OpenAPI 兼容请求链路

现阶段的问题不在协议层，而在管理体验层。当前自定义供应商管理仍然内嵌在设置页 `providers` 分类中，与内置供应商 API Key 配置混在一起，导致以下问题：

1. 页面职责不清晰，内置 API 配置和自定义供应商管理混杂
2. 已添加供应商缺乏可扫描的总览视图，不便于快速定位和维护
3. 新增与编辑都依赖长表单，不符合“管理条目”的操作心智
4. 当前设置页仍保留局部“先编辑、再整体保存”的惯性，不适合供应商这种应立即生效的数据

因此需要将自定义供应商管理从原有设置页内嵌区域中拆出来，形成独立的“供应商”页面，同时保证现有生图运行时逻辑不回归。

## 2. 目标与非目标

### 2.1 目标

本次设计目标如下：

1. 在设置页左侧新增独立菜单项“供应商”，专门用于管理自定义供应商
2. 将原 `providers` 分类外显名称调整为“API 配置”，只保留内置供应商 API Key 和相关说明
3. 将自定义供应商管理改造为“列表页 + 弹窗新增/编辑”的交互模式
4. 新增和编辑操作在弹窗内点击“保存”后立即生效，并立刻刷新供应商列表
5. 列表行直接提供“编辑”和“删除”按钮，其中删除必须经过二次确认
6. 保持现有 `customProviders -> runtimeRegistry -> ModelParamsControls -> Tauri OpenAPI` 运行时链路不变
7. 保证新增、编辑、删除供应商后的结果能够继续反映到生图节点供应商/模型二级选择中

### 2.2 非目标

本次不做以下内容：

1. 不调整 OpenAPI 兼容请求协议格式
2. 不改造 Rust 侧 OpenAPI 兼容执行器
3. 不修改 `customProviders` 的底层数据结构
4. 不将整个设置页统一改造成“即时保存”
5. 不在本次加入“测试连接”“供应商健康检查”“导入导出”“复制供应商”等扩展功能

## 3. 信息架构设计

### 3.1 左侧分类拆分

设置页左侧分类调整为两条清晰职责线：

1. `API 配置`
   - 对应原 `providers` 分类
   - 只保留内置供应商 API Key 配置
   - 保留供应商文档、注册链接、获取 key 指引等说明信息
2. `供应商`
   - 新增独立分类，推荐内部 id 使用 `suppliers`
   - 只用于管理自定义供应商列表和编辑入口

设计原则：

- 用户可见文案使用“API 配置”和“供应商”明确区分职责
- 内部分类 id 不强制重构原 `providers`，避免无谓扩大现有跳转变更范围
- 自定义供应商相关跳转统一指向 `suppliers`

### 3.2 跳转语义

当前项目中已有若干“打开设置页”的入口，后续应按语义拆分：

1. 内置供应商缺少 API Key
   - 继续跳转 `providers`
   - 页面标题显示“API 配置”
2. 自定义供应商未配置或需要管理
   - 跳转 `suppliers`
   - 页面标题显示“供应商”

这样可以避免两个不同问题都被引导到同一个入口，减少用户理解成本。

## 4. 供应商页面交互设计

### 4.1 页面结构

“供应商”页面采用紧凑列表布局，结构如下：

1. 页面头部
   - 标题：`供应商`
   - 简短说明：用于解释这里管理的是自定义供应商与模型
   - 右上角操作按钮：`添加供应商`
2. 列表区域
   - 按供应商条目逐行展示
   - 每行显示供应商名称、已启用模型摘要、右侧操作按钮
3. 空状态
   - 当尚未添加任何自定义供应商时显示空状态提示
   - 仍保留 `添加供应商` 按钮

### 4.2 列表行内容

每个供应商列表行采用紧凑卡片样式，左侧展示信息，右侧展示操作：

左侧：

- 供应商名称
- 已启用模型名称摘要

右侧：

- `编辑`
- `删除`

其中模型摘要规则如下：

1. 只展示已启用模型
2. 多个模型按显示名称拼接
3. 过长时单行截断
4. 如果没有任何已启用模型，则展示“暂无可用模型”

### 4.3 添加供应商弹窗

点击 `添加供应商` 后打开新增弹窗，弹窗内包含当前自定义供应商所需全部字段：

1. 供应商名称
2. 协议类型
   - 当前仅支持 `openapi`
3. `baseUrl`
4. `apiKey`
5. 模型列表
   - 模型显示名称
   - 远程模型 ID
   - 模型启用状态

弹窗底部操作固定为：

- `取消`
- `保存`

点击 `保存` 后执行：

1. 进行字段校验
2. 校验通过则立即写入 `customProviders`
3. 关闭弹窗
4. 刷新供应商列表

本页面不再依赖设置页底部的二次保存。

### 4.4 编辑供应商弹窗

点击列表行 `编辑` 后打开编辑弹窗，结构与新增弹窗保持一致。

区别仅在于：

1. 初始值为当前供应商已保存数据
2. `保存` 后覆盖当前供应商配置
3. 保存成功后立即关闭弹窗并刷新列表

### 4.5 删除交互

点击列表行 `删除` 时，不直接删除，而是弹出二次确认对话框。

确认框设计建议：

1. 标题：`删除供应商`
2. 正文包含供应商名称，例如“确认删除供应商 DChai SG2C 吗？”
3. 按钮：
   - `取消`
   - `确认删除`

点击确认后执行：

1. 立即从 `customProviders` 中移除目标条目
2. 关闭确认框
3. 刷新供应商列表

删除按钮视觉上应弱于编辑按钮，但仍保持可见和可点击。

## 5. 数据流与状态设计

### 5.1 单一真相源

自定义供应商的单一真相源保持不变：

- `src/stores/customProviderConfig.ts`
- `src/stores/settingsStore.ts`

其中核心状态仍为：

```ts
customProviders: CustomProviderConfig[];
```

本次只重构管理入口与交互形式，不修改底层配置结构。

### 5.2 供应商页的保存语义

当前设置页其他分类大多采用“本地编辑 -> 整体保存”的延迟提交方式，但供应商页不再适用该模式。

本次设计建议将供应商页改为即时写入模式：

1. 供应商列表直接读取 store 中的 `customProviders`
2. 添加弹窗维护独立草稿
3. 编辑弹窗维护独立草稿
4. 草稿点击 `保存` 后立即调用 `setCustomProviders(...)`
5. 列表自动重新渲染，无需等待设置页整体保存

这样可以避免出现以下状态冲突：

- 弹窗提示已保存，但页面整体尚未保存
- 用户关闭设置弹窗后丢失供应商编辑结果
- 自定义供应商数据与运行时列表不一致

### 5.3 与运行时链路的兼容关系

以下运行时契约保持不变：

1. `runtimeRegistry` 继续从 `customProviders` 聚合运行时供应商和模型
2. `ModelParamsControls` 继续按“供应商 -> 模型”二级级联展示
3. 生成请求继续透传 `providerRuntime`
4. Rust 侧继续按现有 OpenAPI 兼容链路执行

也就是说，这次页面重构不触碰协议层，只要 `customProviders` 及时更新，生图侧就会继续自动反映最新配置。

### 5.4 删除后的运行时回退

如果用户删除了一个已被节点引用的自定义供应商，不在供应商页面报错，而是沿用当前已有的运行时 fallback 策略：

1. 尝试解析当前模型
2. 若目标模型不存在，则回退到默认可用模型

目标是保证页面侧删除供应商不会让画布节点进入不可恢复状态。

## 6. 错误处理设计

### 6.1 校验范围

弹窗保存前继续复用现有 `validateCustomProviders(...)` 规则，至少覆盖以下场景：

1. 供应商名称为空
2. `baseUrl` 为空
3. `apiKey` 为空
4. 没有任何启用模型
5. 模型显示名称为空
6. 模型远程 ID 为空

### 6.2 校验反馈方式

校验失败时采用弹窗内局部反馈，而不是整页级错误提示：

1. 弹窗保持打开
2. 错误字段高亮
3. 顶部或底部显示简短错误提示
4. 不写入 store

这样可以让错误与操作上下文保持一致，不需要用户回到列表页再理解错误。

### 6.3 删除确认失败与取消

删除场景中：

1. 用户取消确认，则不修改任何数据
2. 只有点击“确认删除”才写入删除结果
3. 删除确认框关闭后，列表状态应保持稳定

## 7. 组件拆分建议

为了避免继续加重 `SettingsDialog.tsx` 的复杂度，建议拆分为以下职责边界：

1. `SettingsDialog`
   - 负责左侧分类切换和整体容器布局
2. `BuiltinProviderSettingsSection` 或等价组件
   - 负责“API 配置”页内容
3. `CustomProvidersPage` 或等价组件
   - 负责“供应商”页列表展示和入口按钮
4. `CustomProviderEditorDialog`
   - 负责新增/编辑供应商表单
5. `DeleteProviderConfirmDialog`
   - 负责删除二次确认

拆分原则：

- 页面容器只做结构
- 列表页只做列表和入口
- 表单弹窗只做草稿编辑和保存
- 确认弹窗只做确认，不混入表单字段

## 8. 测试方案

### 8.1 供应商页测试

需要覆盖以下基础行为：

1. 空状态显示
2. 列表行渲染供应商名称和模型摘要
3. 列表行显示 `编辑` 和 `删除`
4. 点击 `添加供应商` 打开新增弹窗
5. 点击 `编辑` 打开带预填值的编辑弹窗
6. 点击 `删除` 弹出确认框
7. 删除确认后列表立即刷新

### 8.2 弹窗表单测试

1. 缺少必填字段时不能保存
2. 合法数据时保存成功并关闭弹窗
3. 新增后 `customProviders` 立即增加条目
4. 编辑后对应供应商立即更新

### 8.3 设置页分类联动测试

1. 左侧存在“API 配置”和“供应商”两个入口
2. `providers` 分类显示内置 API 配置内容
3. `suppliers` 分类显示自定义供应商列表内容
4. 相关入口跳转到正确分类

### 8.4 生图回归测试

1. 新增供应商后，生图面板可选择该供应商
2. 编辑供应商名称或模型后，生图面板能反映最新状态
3. 删除供应商后，旧节点不崩溃，仍按现有 fallback 逻辑处理

## 9. 风险与取舍

### 9.1 风险

1. 供应商页改为即时保存，而其他设置分类仍是延迟保存，交互语义存在差异
2. 左侧新增分类后，现有 `openSettingsDialog({ category: 'providers' })` 的使用点需要重新梳理
3. 删除按钮进入列表行后，若视觉层级处理不当，容易造成误操作压力

### 9.2 对应取舍

1. 即时保存只限定在供应商页，不扩展到整个设置页
2. 通过分类语义拆分，将内置配置跳转和自定义供应商跳转清楚区分
3. 通过二次确认框降低删除风险

## 10. 实施建议顺序

建议按以下顺序实现：

1. 先扩展设置分类，新增 `suppliers`
2. 将原 `providers` 页面收敛为“API 配置”
3. 新建“供应商”列表页
4. 接入新增/编辑弹窗
5. 接入删除确认弹窗
6. 调整相关入口跳转
7. 最后补齐测试与生图回归验证

这样可以把风险控制在 UI 壳层和页面状态管理中，不影响已完成的 OpenAPI 协议接入能力。
